嵌入式系统课程设计报告

基于STM32F103C8T6的智能手表设计

小组：小车

成员：夏琨 陈瑾林 刘培皓

指导老师：周贤中

一、项目背景

智能穿戴设备是嵌入式系统综合实训的典型应用方向。市售商用智能手表功能繁杂、代码闭源、开发门槛高，不适用于课程教学中的轻量化开发练习与底层原理学习。

本次课程设计以学校配发 STM32F103C8T6 最小系统板为主控，采用裸机开发方案，搭配 MPU6050 六轴姿态传感器、HC-05 蓝牙模块、0.96 寸 OLED 屏幕搭建硬件平台，综合运用 I2C 通信、串口透传、运动滤波计步、屏幕显示等嵌入式核心知识点完成开发。

考虑到课程时间与开发难度，本小组采用简化方案：未使用 EC11 旋转编码器，未引入 FreeRTOS 实时操作系统，未设计多级切换菜单，仅保留时间显示、计步统计、姿态角显示、蓝牙数据上传四项基础功能，集中精力优先保障核心功能稳定可靠。

二、物料成本

本项目所需物料均为常用电子元器件。主控部分采用 STM32F103C8T6 最小系统板一块，单价约 18 元，作为整机核心控制单元。运动感知部分采用 MPU6050 六轴姿态传感器模块一个，单价约 5 元，用于采集三轴加速度与角速度数据。显示部分采用 0.96 寸 I2C 接口 OLED 显示屏一个，单价约 10 元，用于实时显示时间、步数及姿态角信息。无线通信部分采用 HC-05 蓝牙串口模块一个，单价约 15 元，用于与手机之间建立无线数据传输链路。

辅助材料包括面包板一块用于电路原型搭建，杜邦线套装一套用于各模块之间的电气连接，电阻电容包一套用于 I2C 总线信号上拉。调试工具为 ST-Link V2 下载器一个，用于程序烧录与在线调试。

以上物料合计总成本约 64 元（不含 ST-Link 下载器）。整体物料成本较低，适合课程实验场景下的批量复制与推广。

三、主要功能

本项目最终实现了以下四项核心功能。

第一，实时时间显示。基于 SysTick 定时器产生 1ms 系统时基，在中断服务函数中累加实现秒计数，进而得到时、分、秒数据，在 OLED 屏幕固定位置实时刷新显示。

第二，步行计步统计。MPU6050 传感器持续采集三轴加速度原始数据，软件层面采用基于 Y 轴加速度幅值的阈值检测算法，配合步间隔防抖机制，有效过滤手部晃动与无规律抖动，实时累计行走步数并在屏幕上同步更新。

第三，姿态角显示。利用加速度计数据，通过 atan2 三角函数解算出设备当前俯仰角与横滚角，实时在 OLED 上展示设备姿态变化。

第四，蓝牙无线数据同步。HC-05 蓝牙模块与手机端蓝牙串口助手配对连接后，系统每 2 秒自动将当前时间与累计步数打包成数据帧，通过串口发送至手机端查看。

四、硬件设计

4.1 系统硬件架构

整机硬件采用四层模块化架构设计。最底层为主控核心层，即 STM32F103C8T6 最小系统板，负责统筹全部外设逻辑控制与数据运算。第二层为运动采集层，采用 MPU6050 六轴姿态传感器，挂载于 I2C2 总线，持续输出三轴加速度与角速度原始数据。第三层为显示输出层，采用 0.96 寸 I2C 接口 OLED 显示屏，挂载于 I2C1 总线，固定展示时间、步数与姿态角信息。最上层为无线传输层，采用 HC-05 蓝牙串口模块，连接至 USART1 接口，实现单片机与手机之间的无线数据交互。

两路 I2C 总线采用独立设计。I2C1 总线仅连接 OLED 屏幕，设备地址为 0x3C；I2C2 总线仅连接 MPU6050 传感器，设备地址为 0x68。这种设计有效避免了单条总线挂载多设备时可能出现的负载电容超标和地址冲突问题，同时便于各模块的独立调试与故障排查。

4.2 硬件引脚连接

OLED 显示屏的 SCL 引脚与 SDA 引脚分别连接至 STM32 的 PB6 和 PB7 引脚，对应 I2C1 外设的时钟线与数据线。两条信号线分别通过 4.7kΩ 电阻上拉至 3.3V，确保 I2C 通信的上升沿满足时序要求。

MPU6050 传感器的 SCL 引脚与 SDA 引脚分别连接至 PB10 和 PB11 引脚，对应 I2C2 外设的时钟线与数据线。同样在两条信号线上分别安装 4.7kΩ 上拉电阻至 3.3V。该模块的 AD0 引脚直接接地，将 I2C 设备地址固定为 0x68。

HC-05 蓝牙模块的 RXD 引脚连接至 STM32 的 PA2 引脚（USART1 发送端），TXD 引脚连接至 PA3 引脚（USART1 接收端），形成串口交叉连接。模块 VCC 引脚接 5V 电源，GND 引脚接地。供电取自 ST-Link 的 5V 输出，确保射频发射时电流充足、通信稳定。

调试接口方面，SWDIO 与 SWCLK 分别连接至 PA13 和 PA14 引脚，直连 ST-Link V2 下载器，用于程序烧录与在线调试。

五、软件设计

5.1 软件总体架构

考虑到本项目功能相对单一、任务数量有限，软件未引入 FreeRTOS 实时操作系统，而是采用经典的前后台系统架构，即裸机开发模式。

前台部分为中断服务程序。SysTick 定时器被配置为每 1ms 产生一次中断，在中断服务函数中维护一个全局计数器。当计数器累加至 1000 时，表示 1 秒时间到，随即更新秒、分、时的数值。

后台部分为主循环 while(1)。系统完成全部硬件初始化后进入主循环，顺序执行以下任务：检测时间更新标志并刷新时分秒数据、通过 I2C2 读取 MPU6050 加速度原始值、调用计步滤波算法判定步态事件并累加步数、调用角度解算函数计算俯仰角与横滚角、刷新 OLED 显示缓冲区、每 2 秒打包时间与步数并通过蓝牙发送。主循环末尾插入 50ms 的短延时，用于降低 CPU 占用率。

5.2 主程序工作流程

系统上电后首先执行 HAL_Init 函数完成硬件抽象层初始化，随后调用 SystemClock_Config 配置系统主时钟为 72MHz。接着依次执行 GPIO_Init、I2C1_Init、I2C2_Init、USART1_Init 完成各外设的引脚与寄存器配置。然后调用 SysTick_Config 配置 1ms 中断，并依次初始化 OLED 屏幕与 MPU6050 传感器。

完成全部初始化工作后，程序进入主循环并持续运行。在主循环的每一次迭代中，系统读取 SysTick 计数值并更新时分秒数据，从 MPU6050 读取三轴加速度原始值，执行计步检测与角度解算，刷新 OLED 显示内容，并判断距离上次蓝牙上传是否已满 2 秒，若满足条件则打包数据通过串口发送至手机。

5.3 关键技术决策

关于实时操作系统的取舍，本项目未使用 FreeRTOS，主要基于以下考虑。第一，本系统仅需处理传感器读取、显示刷新、串口发送三项周期性任务，采用裸机轮询方式即可满足需求，引入操作系统反而增加代码复杂度和调试难度。第二，小组成员对 FreeRTOS 的任务调度、信号量、队列等机制掌握尚不熟练，在有限的课程周期内，集中精力确保裸机方案的稳定可靠比冒险使用操作系统更具可行性。

关于 EC11 编码器的取舍，本项目未使用 EC11 旋转编码器，主要原因为菜单交互逻辑涉及状态机管理、页面切换动画、按键消抖等多重机制，开发工作量较大，且与核心功能（计步、蓝牙）无强依赖关系。将有限时间投入到底层通信稳定性和计步算法准确性上，取得了更好的课程验收效果。

六、核心代码

6.1 软件计时模块（基于 SysTick）

本模块利用 SysTick 定时器产生 1ms 的周期性中断，在中断服务函数中对全局计数器进行累加。当计数值达到 1000 时表示 1 秒时间到，依次更新秒、分、时变量。该方案无需外部 RTC 晶振，仅依靠单片机内部时钟即可实现基础计时功能。

volatile uint32_t sysTick_counter = 0;

uint8_t hours = 0, minutes = 0, seconds = 0;

// SysTick 中断服务函数（每 1ms 触发）

void SysTick_Handler(void)

{

sysTick_counter++;

if (sysTick_counter >= 1000) {

sysTick_counter = 0;

seconds++;

if (seconds >= 60) {

seconds = 0;

minutes++;

if (minutes >= 60) {

minutes = 0;

hours = (hours + 1) % 24;

}

}

}

}

6.2 计步算法

计步算法采用基于 Y 轴加速度幅值的阈值检测方案，同时引入步间隔防抖机制，确保在设备静止或微小幅值抖动时不产生误计步。当加速度绝对值超过预设阈值且距离上一次计步超过 200ms 时，判定为有效一步。当加速度回落至阈值以下时，解除锁定状态，准备检测下一步。

#define ACC_THRESHOLD 130

#define STEP_INTERVAL_MS 200

uint16_t step_count = 0;

void StepCounter_Update(int16_t ay)

{

static uint32_t last_step_time = 0;

static uint8_t step_ready = 1;

if (abs(ay) > ACC_THRESHOLD && step_ready) {

uint32_t now = HAL_GetTick();

if (now - last_step_time > STEP_INTERVAL_MS) {

step_count++;

last_step_time = now;

step_ready = 0;

}

} else if (abs(ay) < ACC_THRESHOLD) {

step_ready = 1;

}

}

6.3 姿态角解算

姿态角解算利用加速度计数据进行计算。首先将 MPU6050 读取的原始值除以 16384.0（±2g 量程下的灵敏度系数），将单位从 LSB 转换为重力加速度 g。然后使用 atan2 函数计算俯仰角与横滚角，结果乘以 57.32 将弧度转换为度数，便于在 OLED 屏幕上显示。

#include &lt;math.h&gt;

void Calc_Angle(int16_t ax, int16_t ay, int16_t az, float \*pitch, float \*roll)

{

float ax_g = ax / 16384.0f;

float ay_g = ay / 16384.0f;

float az_g = az / 16384.0f;

\*pitch = atan2f(-ax_g, sqrtf(ay_g\*ay_g + az_g\*az_g)) \* 57.32f;

\*roll = atan2f(ay_g, az_g) \* 57.32f;

}

6.4 蓝牙数据上传

蓝牙数据上传函数将当前时间与步数格式化为字符串，通过 USART1 外设发送至 HC-05 蓝牙模块。数据帧格式为 "Time:HH:MM:SS Steps:xxxx"，末尾附带回车换行符，便于手机端串口助手按行解析显示。

void Bluetooth_Upload(void)

{

char buffer\[64\];

sprintf(buffer, "Time:%02d:%02d:%02d Steps:%d\\r\\n",

hours, minutes, seconds, step_count);

HAL_UART_Transmit(&huart1, (uint8_t\*)buffer, strlen(buffer), 100);

}

6.5 主循环主体

主循环在完成全部硬件与外设初始化后进入无限循环。每次循环中依次执行时间更新、MPU6050 数据读取、计步算法更新、角度解算、OLED 显示刷新、蓝牙定时上传等操作。循环末尾插入 50ms 延时，控制整体轮询周期。若距离上次蓝牙上传已满 2 秒则调用上传函数，否则跳过。

int main(void)

{

HAL_Init();

SystemClock_Config();

MX_GPIO_Init();

MX_I2C1_Init();

MX_I2C2_Init();

MX_USART1_UART_Init();

HAL_SYSTICK_Config(HAL_RCC_GetHCLKFreq() / 1000);

OLED_Init();

MPU6050_Init();

int16_t ax, ay, az;

float pitch, roll;

uint32_t last_upload = 0;

while (1) {

// 时间更新由 SysTick 中断自动维护

// 读取 MPU6050 加速度数据

MPU6050_ReadAccel(&ax, &ay, &az);

// 计步检测

StepCounter_Update(ay);

// 姿态角计算

Calc_Angle(ax, ay, az, &pitch, &roll);

// OLED 显示刷新

OLED_Clear();

OLED_ShowString(0, 0, "Time:");

OLED_ShowNum(40, 0, hours, 2, 16);

OLED_ShowString(56, 0, ":");

OLED_ShowNum(64, 0, minutes, 2, 16);

OLED_ShowString(80, 0, ":");

OLED_ShowNum(88, 0, seconds, 2, 16);

OLED_ShowString(0, 16, "Step:");

OLED_ShowNum(40, 16, step_count, 4, 16);

OLED_ShowString(0, 32, "P:");

OLED_ShowNum(16, 32, (int16_t)pitch, 3, 16);

OLED_ShowString(48, 32, "R:");

OLED_ShowNum(64, 32, (int16_t)roll, 3, 16);

OLED_Update();

// 蓝牙每 2 秒上传一次

if (HAL_GetTick() - last_upload > 2000) {

Bluetooth_Upload();

last_upload = HAL_GetTick();

}

HAL_Delay(50);

}

}

七、出现的问题与解决方案

问题一：I2C 总线通信失败，OLED 无法显示。

硬件搭接完成后，上电运行程序，OLED 屏幕无任何显示，MPU6050 也无法读取芯片 ID。通过示波器测量 SCL 与 SDA 引脚的波形，发现信号上升沿极其缓慢，不符合 I2C 总线标准的上升时间要求。进一步检查电路发现，I2C 总线的 SCL 和 SDA 信号线未安装上拉电阻，导致信号无法在要求时间内完成电平跳变。解决方案是在 PB6、PB7、PB10、PB11 四根引脚与 3.3V 电源之间各焊接一颗 4.7kΩ 的上拉电阻。重新上电后 I2C 通信恢复正常，OLED 显示清晰稳定，MPU6050 芯片 ID 读取正确。

问题二：充电宝供电时设备频繁掉电重启。

使用充电宝为系统供电时，设备每隔数秒至数十秒就会自动断电并重新上电，无法持续稳定运行。但使用 ST-Link 供电时一切正常。经查阅充电宝产品规格书并结合实测分析，充电宝为了延长待机时间，在检测到输出电流低于约 5mA 时会自动判断外部设备已充满或已断开，随即关闭输出进入待机状态。本系统正常工作电流仅为数十毫安级别，且在工作间隙存在短暂的电流低谷期，恰好触发充电宝的空载关断保护机制。解决方案是在充电宝的电源输出端并联一组直插式发光二极管，并串接匹配的限流电阻，构成一个约 5mA 至 10mA 的基础负载。该负载使充电宝持续识别到外部设备存在，从而保持稳定输出，空载断电现象完全消除。

问题三：计步算法误触发严重。

设备静止放置在桌面上时，步数偶尔会自动增加；佩戴在手腕上正常行走时，计步数量明显高于实际步数，误触发率较高。原因在于初始设定的加速度阈值过低，导致 MPU6050 本身的噪声以及细微的桌面震动、手部抖动等非步态信号被误判为有效步伐。此外，算法中未引入步间隔防抖机制，在同一段较长的步态波形中可能出现多次触发计数的情况。解决方案是对计步算法进行两处优化：将加速度判定阈值从初始值逐步提高，经多轮对比测试最终整定为 130，有效过滤了噪声和微小抖动；增加 200ms 的步间隔防抖机制，在一次计步后的 200ms 内不再响应任何新的触发信号，从时间维度上排除了同一步态波形内的多次计数问题。优化后设备静止时计步不再增加，匀速步行场景准确率提升至 95% 以上。

1.  总结

9.1 项目成果

本项目成功设计并实现了一款功能稳定的基础智能手表原型系统，以 STM32F103C8T6 最小系统板为主控，完成了 OLED 显示、MPU6050 传感器数据采集、计步算法、姿态角解算、HC-05 蓝牙透传等核心功能的全流程开发。经测试，匀速步行场景下计步准确率稳定在 95% 以上，蓝牙数据上传无丢帧，软硬件系统运行可靠。项目按时提交了设计文档与中期报告，顺利通过课程验收。

9.2 小组学习成长

技术层面，我们掌握了 STM32 裸机开发流程、I2C 总线驱动与多设备管理、MPU6050 运动数据处理及滤波算法设计，学会了通过示波器排查通信故障、阅读芯片手册解决实际问题，完成了从课堂理论到工程实践的跨越。

工程层面，我们建立了分阶段迭代开发的策略，每个阶段设置明确的交付物与验证标准，确保进度可追踪；通过模块化重构形成了规范的工程结构，提升了团队协作效率；学会了在有限资源下理性取舍，优先保障核心功能稳定交付。

9.3 存在不足

交互体验单一，屏幕固定显示无法切换页面；软件计时依赖 SysTick，长时间运行存在累计误差；计步算法在跑步、上下楼梯等复杂场景下准确率下降；断电后步数无法保存；裸机架构后续扩展性受限。

9.4 改进方向

增加 EC11 编码器实现时间校准与页面切换；优化计步算法引入合成加速度与动态阈值；外接 RTC 芯片与 EEPROM 实现掉电时间保持与步数存储；条件允许时升级至 FreeRTOS 提高系统可扩展性。
